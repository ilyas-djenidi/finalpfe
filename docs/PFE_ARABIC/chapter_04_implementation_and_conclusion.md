
# الفصل الرابع: التنفيذ والإنجاز

---

## 1. مقدمة

وصلنا إلى الفصل الأهم من الناحية العملية: عرض ما بنيناه فعلاً. في هذا الفصل نُقدّم واجهات النظام صفحة بصفحة، ونشرح كيف يعمل كل جزء من التطبيق في الواقع. ثم نعرض الأدوات والتقنيات الكاملة التي اعتمدنا عليها في بناء المشروع من الصفر.

---

## 2. عرض النظام

### 2.1 صفحة الدخول والتسجيل

صفحة الدخول هي أول ما يراه المستخدم — وقد صمّمناها بأسلوب **Cyberpunk** مناسب لطبيعة المنصة الأمنية: خلفية داكنة، ألوان نيون زرقاء، وحدود متوهجة تُعطي إحساساً بالتقنية.

**من الناحية الأمنية:**
- حقل كلمة المرور يملك شروطاً واضحة (10 أحرف على الأقل، أرقام، أحرف كبيرة وصغيرة، رموز خاصة)
- كل نموذج يحمل رمز CSRF يتغير مع كل جلسة
- تشفير bcrypt بـ 12 جولة يحمي كلمات المرور حتى في حالة تسريب قاعدة البيانات

---

### 2.2 لوحة التحكم الرئيسية (Dashboard)

لوحة التحكم تعطي نظرة عامة فورية عن الوضع الأمني:

**المعلومات المعروضة:**
- **إجمالي الفحوصات المُنجزة** مع تصنيفها حسب النوع
- **توزيع الثغرات** حسب مستوى الخطورة (مخطط دائري)
- **أعلى 5 أهداف خطورةً** فُحصت مؤخراً
- **آخر نشاط** مع التوقيت

---

### 2.3 ماسح الويب (Web Scanner)

هذه الصفحة هي قلب المنصة. المستخدم يُدخل URL الهدف ويضغط "Scan" — والنظام يبدأ تلقائياً:

**مراحل الفحص المرئية للمستخدم:**
1. ✅ التحقق من الاتصال بالهدف
2. ✅ استكشاف صفحات الموقع (Spidering)
3. ✅ فحص OWASP A01–A10 بالتوازي
4. ✅ الفحوصات الإضافية (SQLi، SSRF، Log4Shell...)
5. ✅ توليد التقرير

**صفحة النتائج تُظهر:**
- عدد الثغرات المكتشفة مُصنَّفة حسب الخطورة
- تفاصيل كل ثغرة: الوصف، OWASP ID، CWE، درجة CVSS، والدليل
- لون مميز لكل مستوى خطورة: 🔴 حرجة، 🟠 عالية، 🟡 متوسطة، 🟢 منخفضة

**أزرار الإجراء:**
- "تحليل بالذكاء الاصطناعي" → تقرير تنفيذي كامل من Gemini
- "تحميل التقرير" → تحميل ملف MD أو CSV أو JSON

---

### 2.4 ماسح إعدادات Apache

المستخدم يرفع ملف `httpd.conf` أو `.htaccess`، والنظام يُحلّله على الفور.

**مثال على نتيجة فحص حقيقية:**

```
[❌ CRITICAL] كلمة مرور واضحة في ملف الإعداد
الوصف: تم اكتشاف كلمة مرور مكتوبة بنص واضح داخل الملف
CWE: CWE-798 — Use of Hard-coded Credentials
CVSS: 9.1 / Critical
الحل: استخدم متغيرات البيئة (Environment Variables)

[⚠️ HIGH] بروتوكول TLS 1.1 مُفعَّل
الوصف: بروتوكول TLS 1.1 به ثغرات معروفة ولم يعد مدعوماً
CWE: CWE-327 — Use of Broken Algorithm
CVSS: 7.5 / High
الحل: SSLProtocol TLSv1.2 TLSv1.3
```

**الميزة الحصرية — "تحميل الإعداد المُصحَّح":**
بعد الفحص، يمكن تحميل ملف `httpd.conf` جديد مُصحَّح تلقائياً، مع تعليق توضيحي عند كل سطر مُعدَّل.

---

### 2.5 محلل الكود المصدري (SAST)

المستخدم يرفع ملف ZIP يحتوي الكود (PHP، Python، JavaScript، أو Java).

**مثال على نتائج فحص كود PHP:**

```
[❌ CRITICAL] SQL Injection — السطر 47
الكود المشكل:
    $query = "SELECT * FROM users WHERE id = " . $_GET['id'];
الحل الآمن:
    $stmt = $pdo->prepare("SELECT * FROM users WHERE id = ?");
    $stmt->execute([$_GET['id']]);

[⚠️ HIGH] XSS — السطر 83
الكود المشكل:
    echo "مرحباً " . $_GET['name'];
الحل الآمن:
    echo "مرحباً " . htmlspecialchars($_GET['name'], ENT_QUOTES, 'UTF-8');
```

**ميزة "إصلاح الكود":**
اضغط الزر — وسيُعيد Gemini كتابة الملف كاملاً مع كل الإصلاحات مطبّقة وشرح لكل تعديل.

---

### 2.6 ماسح الشبكة (Network Scanner)

المستخدم يُدخل عنوان IP أو نطاق CIDR (مثلاً `192.168.1.0/24`).

**المعلومات التي يكشفها الفحص:**
- **معلومات الهدف:** الموقع الجغرافي، رقم الـ AS، إصدار نظام التشغيل (بالاستدلال)
- **المنافذ المفتوحة:** مع اسم الخدمة وإصدارها
- **الثغرات المكتشفة:** FTP مجهول، SSH بخوارزميات ضعيفة، RDP مكشوف، إلخ

**مثال على نتيجة:**
```
🔴 [CRITICAL] Redis بدون مصادقة — المنفذ 6379
    قاعدة بيانات Redis مكشوفة للعموم بدون كلمة مرور
    → يسمح بقراءة وكتابة جميع البيانات المخزّنة
    CVE-2022-0543 — CVSS: 10.0

🟠 [HIGH] SSH يستخدم خوارزمية ضعيفة — المنفذ 22
    diffie-hellman-group1-sha1 معروف بضعفه منذ 2016
    → الحل: تعطيل الخوارزميات الضعيفة في /etc/ssh/sshd_config
```

---

### 2.7 صفحة التقارير

صفحة مركزية تُعرض فيها كل فحوصات المستخدم السابقة (في النسخة المستقبلية مع قاعدة البيانات). كل تقرير يمكن:
- استعراضه مباشرةً في المتصفح
- تحميله بصيغة Markdown (للمطورين)
- تحميله بصيغة CSV (للإدارة والجداول)
- تحميله بصيغة JSON (للتكامل مع أنظمة أخرى)

### 2.8 ChatBot الأمني

```
المحلل: ما هو الفرق بين XSS المنعكس والمخزون؟

SecuraX: 
XSS المنعكس (Reflected XSS): يعمل في طلب واحد — المهاجم يُرسل 
رابطاً يحتوي الكود الخبيث، وعندما يفتح الضحية الرابط يُنفَّذ 
الكود في متصفحه مباشرةً. لا يُخزَّن في الخادم.

XSS المخزون (Stored XSS): أخطر بكثير — الكود الخبيث يُحفظ في 
قاعدة بيانات الموقع (في تعليق أو منشور مثلاً)، ويُنفَّذ عند 
كل مستخدم يزور الصفحة. يمكنه التأثير على آلاف المستخدمين.

مثال على منع الاثنين معاً في PHP:
    echo htmlspecialchars($user_input, ENT_QUOTES, 'UTF-8');
```

الـ ChatBot يعرف سياق آخر فحص أجراه المستخدم، فيمكنه الإجابة على أسئلة خاصة بالنتائج المكتشفة.

## 4. الخاتمة

هذا الفصل قدّم الصورة الكاملة لما بنيناه: من صفحات الواجهة التي يراها المستخدم، إلى التقنيات التي تعمل خلف الكواليس. المشروع جاهز للاستخدام الفعلي — يكفي أن تُدخل URL أي موقع وتنتظر دقيقتين لتحصل على تقرير أمني كامل يُعادل ما كان يكلّف آلاف الدولارات من قبل.

# الخاتمة العامة

عندما بدأنا العمل على هذا المشروع، كان لدينا سؤال واحد بسيط: **لماذا يجب على المطور الصغير أو الطالب أن يعيش في عمى أمني؟** الأدوات موجودة، لكنها مبعثرة ومعقدة ومكلفة. والفراغ بين من يملك Nessus وBurp Suite بميزانيات ضخمة، وبين من لا يملك شيئاً — هذا الفراغ هو المكان الذي بنيناه لـ **SecuraX**.

خلال هذه الرحلة، لم نكتفِ بتجميع أدوات موجودة — بل أضفنا طبقة ذكاء حقيقية تُحوّل بيانات الثغرات الجافة إلى تقارير مفهومة يتصرف بناءً عليها المطور والمدير والمراجع الخارجي على حد سواء. وعدنا بخوارزميات مبتكرة لكشف SQLi تتجاوز حدود الطرق التقليدية، وبنظام معماري يعمل حتى وقت انقطاع الإنترنت.

**ما أنجزناه:**
- منصة كشف ثغرات متكاملة تغطي 4 طبقات (الويب، الكود، الشبكة، الخادم)
- تغطية كاملة لـ OWASP Top 10 2025 بخوارزميات مخصصة
- دمج ذكاء اصطناعي Gemini مع وضع احتياطي محلي دائم العمل
- واجهة مستخدم احترافية وسهلة تُلغي الحاجة لخبرة أمنية عميقة

**ما نطمح إليه مستقبلاً:**
- مراقبة مستمرة مع تنبيهات تلقائية عند ظهور ثغرات جديدة
- تكامل مع CI/CD لإيقاف النشر عند اكتشاف ثغرات حرجة
- نموذج تعلم آلي مُدرَّب على أنماط الثغرات لكشف الأنماط الجديدة
- تطبيق جوال يُتيح الفحص من أي مكان

الأمن الإلكتروني ليس ترفاً. هو حق لكل مطور، ولكل موقع، ولكل مؤسسة مهما كان حجمها. وهذا ما نحاول أن نُحققه مع **SecuraX**.
